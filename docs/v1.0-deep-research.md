# v1.0 深度研究闭环

## 目标

让复杂问题不再被压缩成一次即时问答。用户提交课题后，系统在后台自主规划、检索、
验证和写作，即使关闭页面也不会终止，最终交付可追溯、可导出的研究报告。

## 用户流程

1. 在左侧“深度研究”中新建任务，填写课题。
2. 选择一个或多个知识库，并决定是否使用长期记忆和联网搜索。
3. 后台生成 3～6 个可独立检索的子问题。
4. 每个子问题从启用的数据源收集证据，并形成受证据约束的研究笔记。
5. Verifier 检查覆盖度、缺口和冲突；证据不足时追加有限的补充检索。
6. 系统生成包含执行摘要、研究发现、风险、不确定性、结论和参考资料的 Markdown
   报告。
7. 用户可查看计划、步骤、证据和报告，失败任务可以重新运行。

智能问答输入框旁提供“深度研究”入口，会把当前问题、已选知识库和联网开关带到新建
研究表单。两种能力入口互通，但数据模型和执行生命周期彼此独立。

## 数据模型

- `research_tasks`：课题、数据源设置、阶段、进度、研究计划、Verifier 结果、报告、
  Token 和错误状态。
- `research_steps`：规划产生的子问题以及后续补充问题，保存执行状态、研究笔记和
  证据数量。
- `research_evidence`：来源类型、来源标识、片段、标题、引文、URL、定位信息、
相关度和产生它的检索问题。

所有模型直接绑定 `user_id` 或归属于用户隔离的研究任务。

## 工作流

```text
queued
  → planning
  → collecting_evidence
  → verifying_evidence
      ├─ 证据不足且未达到上限 → collecting_evidence
      └─ 证据充分或达到上限 → writing_report
  → completed
```

研究计划最多六个初始问题，Verifier 最多运行三轮，每轮最多追加三个问题。证据按来源
和片段去重，每个任务最多保存 40 条进入报告上下文；引文进入模型前还会截断，避免
上下文无界增长。

最终写作后还会执行确定性引用校验：只有落在实际证据编号范围内的 `[n]` 才能保留。
当证据库为空时，系统直接生成“证据不足”报告，不让模型生成占位引用。

## 可靠执行

- 创建研究任务和 `task_outbox` 在同一个 PostgreSQL 事务中提交。
- Celery 使用独立 `research` 队列，任务开启晚确认和 Worker 丢失重投。
- 临时错误按指数退避重试；失败后前端提供手动重试。
- Beat 检查长时间停留在规划、研究、验证和写作阶段的任务，并重新排队。
- 删除任务会同时清理步骤、证据和对应 Outbox 记录。

## API

```text
POST   /api/research
GET    /api/research
GET    /api/research/{task_id}
POST   /api/research/{task_id}/retry
GET    /api/research/{task_id}/export
DELETE /api/research/{task_id}
```

前端每 2.5 秒轮询仍在运行的任务；完成后停止轮询。

## Token 原则

研究任务累计规划、研究笔记、Verifier 和报告生成的输入、输出及总 Token。系统不维护
模型价格，不估算或保存费用。

## 验收

- 数据库迁移可正向执行且 Alembic 无模型漂移。
- 研究任务和 Outbox 在真实 PostgreSQL 中原子创建。
- 规划解析、补充问题、证据编号、上下文上限和 Token 累计具有单元测试。
- 前端生产构建通过。
- Worker 能注册 `app.tasks.research.run_research` 并监听 `research` 队列。
