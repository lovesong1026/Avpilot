# v0.9 质量评测与可观测性

## 目标

让 Avpilot 不仅“能回答”，还可以解释一次回答经历了什么、用了多少 Token、检索到
什么、哪里失败，以及新版本是否让 RAG 质量退化。

本版本明确不维护模型价格，不估算或保存任何费用。价格随供应商、区域、套餐和时间
变化，当前产品决策只需要稳定的 Token 与耗时事实。

## Agent Trace

每轮问答在用户消息持久化后创建 Trace：

```text
AgentTrace
├── AgentSpan: agent_orchestration
├── AgentSpan: knowledge_search / memory_search / web_search
├── RetrievalSnapshot
│   ├── query
│   ├── hit_count / top_score
│   └── citations
├── ModelUsage: agent_planning
├── AgentSpan: answer_generation
└── ModelUsage: answer_generation
```

Trace 与用户、会话、用户消息和助手消息关联。SSE 的 `meta` 与 `completed` 事件均
返回 `trace_id`。发生异常时 Trace 会进入 `failed`，记录异常类型和安全截断后的
错误信息。

## 数据职责

### `agent_traces`

- 问题、会话和消息归属
- 状态、Agent 模式、总耗时
- 工具次数、引用次数和失败原因

### `agent_spans`

- Agent 编排、单次工具执行和最终回答三个层次
- 开始、结束、耗时、状态
- 有界的输入和输出摘要，不保存模型密钥或隐藏推理

### `model_usages`

- 百炼模型名称和调用阶段
- 输入、输出、总 Token
- 调用耗时和状态
- 不含单价、币种和费用字段

### `retrieval_snapshots`

- 工具调用 ID、工具、查询、命中数和耗时
- 引用的来源、片段、定位和分数
- 用于回答复盘和离线评测样本构建

## API

```text
GET /api/observability/summary?days=14
GET /api/observability/traces?limit=30
GET /api/observability/traces/{trace_id}
```

三个接口都从当前 JWT 用户派生归属，不接受外部 `user_id`。

## 仪表盘

仪表盘新增：

- 近 14 天输入/输出 Token 趋势
- Agent 成功率和平均总耗时
- 工具调用次数与工具分布
- 最近 Trace 列表
- Trace 阶段时间线、模型 Token 和检索快照抽屉

已有知识、记忆和标签图表继续保留。

## RAG 离线评测

评测集使用版本化 JSONL，每个案例包含问题、期望文档 ID 或标题，以及可选参考答案。
执行器调用真实混合检索并计算：

- Recall@K
- MRR
- 引用文档命中率
- 引用编号有效率
- 无来源时的伪造引用标记

评测指标是确定性计算，不调用 LLM 充当裁判。这样评测不会额外消耗 Token，也不会因
裁判模型变化产生不可解释波动。

运行方法见 [eval/README.md](../eval/README.md)。

## 自动质量门禁

`.github/workflows/quality.yml` 包含四个 Job：

1. 后端：uv 锁文件、Ruff、非集成测试。
2. 前端：`npm ci`、TypeScript 和 Vite 生产构建。
3. 集成：真实 PostgreSQL/Redis、Alembic 升级与漂移检查、Trace/Outbox 数据库
   往返、Celery 任务往返。
4. Compose：验证环境样例可以生成合法 Compose 配置。

## 验收

- 每次成功问答均能通过 `trace_id` 查询完整记录。
- 失败问答不会留下永久 `running` Trace。
- 最终回答 Token 与助手消息返回的原始 usage 一致。
- 用户不能读取其他用户的 Trace。
- 仪表盘无数据时返回零值，不报错。
- Alembic 模型与迁移无漂移。
- RAG 指标函数有固定测试样例。
- GitHub 推送和 PR 自动运行质量检查。
