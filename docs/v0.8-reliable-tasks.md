# v0.8 可靠任务系统

## 目标

把文档、网页、图片和记忆萃取从 FastAPI 进程迁移到可恢复的后台任务层。系统采用
“至少一次投递”，优先保证任务不会静默丢失，再通过任务认领和写入幂等消除重复执行
造成的副作用。

实时 Agent 问答继续留在 FastAPI 内通过 SSE 执行；回答结束后的记忆萃取才进入
后台队列。

## 运行架构

```text
React ──HTTP/SSE──> FastAPI
                       │
             同一 PostgreSQL 事务
                       │
              业务记录 + TaskOutbox
                       │
                       ▼
                 Redis broker
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   ingestion        memory       maintenance
   文档/网页/图片     记忆萃取       补投/恢复/聚类
        │              │
        ▼              ▼
 Elasticsearch       Neo4j
```

PostgreSQL 是任务进度和业务状态的事实源。Celery result backend 只保留短期运行结果，
前端不依赖它展示状态。

## Transactional Outbox

上传或主动记忆时，API 在同一个数据库事务中写入业务记录和 `task_outbox`：

```text
BEGIN
  INSERT document / image / memory_source
  INSERT ingestion_job
  INSERT task_outbox
COMMIT
```

提交后立即尝试投递 Redis。投递失败不会回滚已经保存的文件或业务记录，Outbox 会
进入 `dispatch_failed`，Beat 每 30 秒扫描并补投。这样避免“数据库提交成功但消息
没有发出去”的双写裂缝。

`dedupe_key` 对每个处理周期唯一。用户手动重试会复用并重置原 Outbox，不重复创建
相同任务意图。

## 可靠性策略

- `task_acks_late=True`：任务完成后才确认消息。
- `task_reject_on_worker_lost=True`：Worker 异常退出时重新投递。
- `worker_prefetch_multiplier=1`：长任务不在单个 Worker 中囤积。
- Redis `visibility_timeout=3600`：未确认任务在窗口后重新可见。
- 文档任务最长 21 分钟，图片和记忆任务最长 16 分钟。
- 临时错误最多自动重试四次，退避为 30、60、120、240 秒。
- 文件损坏、不支持格式、SSRF 安全拦截和文件不存在等永久错误不自动重试。
- Outbox 使用 PostgreSQL 行锁原子认领；已完成或正在执行的普通重复消息直接跳过。
- Worker 丢失产生的 `redelivered` 消息允许重新认领原任务。
- 处理超过 30 分钟仍未更新状态时，Beat 将其判定为中断并恢复；超过最大尝试次数
  后进入最终失败。

## 幂等边界

- 文档和网页：写 ES 前按文档 ID 删除旧分块，再批量写入当前版本。
- 图片：写 ES 前按图片 ID 删除旧语义索引。
- 记忆：片段 ID 由来源、顺序和内容确定，Neo4j 使用稳定业务键和 `MERGE` 复用
  来源、片段、陈述及实体。
- Outbox：同一个任务周期使用唯一 `dedupe_key` 和稳定 Celery task ID。

PostgreSQL、Elasticsearch 和 Neo4j 不组成分布式事务。ES 和 Neo4j 被视为可重建
派生数据，通过幂等任务实现最终一致。

## 状态与恢复

入库状态：

```text
pending → processing → completed
   │           │
   │           └→ retrying → processing
   └→ dispatch_failed → queued
                         └→ failed → 用户手动重试
```

前端轮询 PostgreSQL 中的文档、图片和记忆状态。失败卡片提供“重试”按钮，重新排队
时清除旧错误并开始新的自动重试周期。

## 启动

```bash
docker compose up -d postgres elasticsearch neo4j redis
cd api
uv sync
uv run alembic upgrade head
cd ..
docker compose up -d worker beat
```

开发时也可以在两个终端中运行：

```bash
cd api
uv run celery -A app.worker.celery_app worker -l INFO \
  -Q ingestion,memory,maintenance --concurrency=2
```

```bash
cd api
uv run celery -A app.worker.celery_app beat -l INFO
```

macOS 如果 prefork 调试遇到第三方原生库限制，可临时追加 `--pool=solo`；生产 Linux
继续使用默认 prefork。

## 验收

- Redis 不可用时上传仍会留下可恢复 Outbox，Redis 恢复后自动补投。
- FastAPI 重启不影响 Worker 中已经开始的任务。
- Worker 被强制终止后，未确认消息会重新执行。
- 同一个 Celery 消息重复投递不会并发执行两次。
- 百炼、ES、Neo4j 暂时超时会自动退避重试。
- 永久输入错误不会无限重试。
- 用户能看到排队、处理、重试、成功、失败，并能手动重试。
